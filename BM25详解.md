# BM25 详解：搜索引擎的"打分神器"

> 写给零基础读者，尽量少用术语，多用比喻。

---

## 一、它是什么？

**BM25** 全称 *Best Matching 25*（"最佳匹配第 25 版"，25 是论文的迭代版本号）。

简单一句话：**BM25 是一个"打分公式"，用来衡量一篇文档和一句搜索词有多相关。**

### 先看一个生活中的场景

你在搜东西时（比如搜淘宝、搜网页），系统需要做两步：

1. **召回**：从几亿个文档里，找出"可能相关"的几百篇。
2. **排序**：把这几百篇按"相关性"从高到低排好，最相关的放最上面。

BM25 干的就是**第 2 步——打分排序**。

> 打个比方：图书馆管理员收到你的需求"找讲机器学习的书"，他需要从书架上挑出几十本可能相关的，然后**按"有多符合你的需求"打分排序**。BM25 就是这位管理员的"打分规则手册"。

---

## 二、为什么需要 BM25？（先看看它解决了什么问题）

### 最朴素的思路：数关键词出现次数

比如搜 `"苹果 手机"`，A 文档里"苹果"出现 10 次，B 文档里"苹果"出现 3 次，那 A 更相关？

**但朴素方法有几个大坑：**

#### 坑 1：词频"刷榜"
一个文档疯狂重复某个词 100 次，并不意味着它更相关。真正好的文档可能提 3 次就够了。

> BM25 的解法：**词频"饱和效应"**——词出现越多，加分越少（像吃包子：第一个最香，吃到第十个就腻了）。

#### 坑 2：常见词没价值
"的"、"是"、"the"、"a" 这种词到处都是，搜它毫无意义。即使是"苹果"，如果每篇文章都有，也不算稀奇。

> BM25 的解法：**IDF（逆文档频率）**——越稀有的词，权重越高（"独门秘籍"比"基本功"更值钱）。

#### 坑 3：长文档"占便宜"
一篇 10000 字的文档，自然比 100 字的短文档更容易出现某个词。但长不代表更相关。

> BM25 的解法：**文档长度归一化**——按比例打折，让长短文档公平竞争。

---

## 三、BM25 公式（别怕，我们一步步拆）

完整公式看起来有点吓人：

```
score(文档, 查询) = Σ IDF(词) × [ tf × (k1 + 1) ] / [ tf + k1 × (1 - b + b × 文档长度/平均长度) ]
```

`Σ` 表示对查询里**每个词**分别算一遍分数，然后**加起来**。

### 我们拆成 3 个零件来看

#### 零件 1：IDF —— 这个词有多"稀有"

```
IDF(词) = ln [ (文档总数 - 包含该词的文档数 + 0.5) / (包含该词的文档数 + 0.5) + 1 ]
```

- 包含这个词的文档**越少** → IDF **越大** → 这个词越**值钱**。
- 所有人都有的词 → IDF 接近 0 → 几乎不加分。

> 比喻：在一个村子里，"会做饭"几乎人人都会（IDF 低，不值钱），但"会开直升机"只有 1 个人会（IDF 极高，非常值钱）。

#### 零件 2：TF 饱和 —— 出现次数怎么算分

```
tf 部分 = tf × (k1 + 1) / (tf + k1 × ...)
```

- `tf` = 词在文档里出现了几次
- `k1` 是一个**调节"饱和速度"的旋钮**（通常取 `1.2 ~ 2.0`）

效果：
- 出现 1 次 → 加很多分
- 出现 3 次 → 再加一点
- 出现 100 次 → 几乎和出现 10 次一样了

> 比喻：吃第 1 个包子很满足，第 3 个还行，第 10 个撑了，第 100 个……再多也没区别。

#### 零件 3：文档长度归一化 —— 长短公平

```
长度因子 = (1 - b + b × 文档长度 / 平均文档长度)
```

- `b` 是**调节"长度惩罚力度"的旋钮**（通常取 `0.75`）
- 文档比平均**长** → 分母变大 → 分数**被打折**
- 文档比平均**短** → 几乎不打折

> 比喻：考试改卷时，老师会考虑"答题篇幅"。同样的关键词，写在 200 字短文里出现的，比写在 10000 字长文里偶然出现的，更"实诚"。

### 两个旋钮总结

| 参数 | 默认值 | 作用 | 调大会怎样 |
|------|--------|------|------------|
| `k1` | 1.2 ~ 2.0 | 控制词频饱和快慢 | 词频影响更明显（刷词更容易） |
| `b`  | 0.75    | 控制长度惩罚力度  | 长文档被打折得更狠 |

## 四、一个完整的计算例子

假设：
- 文档库共 **1000 篇** 文档，平均长度 **200 词**
- 搜 `"苹果"`，其中 **100 篇** 含"苹果"
- 文档 A 长度 **400 词**，"苹果"出现 **5 次**

**第 1 步：算 IDF**

```
IDF("苹果") = ln [ (1000 - 100 + 0.5) / (100 + 0.5) + 1 ]
            = ln [ 900.5 / 100.5 + 1 ]
            = ln [ 9.96 ]
            ≈ 2.30
```

**第 2 步：算 TF 部分（取 k1=1.5）**

```
tf 部分 = 5 × (1.5 + 1) / (5 + 1.5 × ...)
```

**第 3 步：算长度因子（取 b=0.75）**

```
长度因子 = 1 - 0.75 + 0.75 × (400 / 200)
         = 0.25 + 0.75 × 2
         = 1.75
```

**第 4 步：组合（分母里要加上长度因子）**

```
score = 2.30 × [ 5 × 2.5 / (5 + 1.5 × 1.75) ]
      = 2.30 × [ 12.5 / 7.625 ]
      ≈ 2.30 × 1.64
      ≈ 3.77
```

文档 A 关于"苹果"的 BM25 分数 ≈ **3.77**。同理给其他文档打分，**分数高的排前面**。

## 五、怎么实现的？（伪代码 + Python）

### 伪代码（理解逻辑）

```text
1. 离线预处理（建索引时做一次）：
   - 统计每篇文档的长度 → 算出平均长度 avgdl
   - 对每个词，统计有多少篇文档包含它 → 算 IDF
   - 建一个倒排索引：词 → 出现在哪些文档、各出现几次

2. 在线查询（用户搜索时）：
   - 切分查询词
   - 对每个查询词：
       从倒排索引拿到"包含这个词的文档列表 + 词频"
       对每个候选文档，套 BM25 公式算分
   - 把所有词的分数相加
   - 按总分排序，返回 Top N
```

### Python 最简实现

```python
import math
from collections import Counter

class BM25:
    def __init__(self, documents, k1=1.5, b=0.75):
        self.k1 = k1
        self.b = b
        self.docs = [doc.split() for doc in documents]      # 简单按空格切词
        self.doc_len = [len(doc) for doc in self.docs]
        self.avgdl = sum(self.doc_len) / len(self.doc_len)  # 平均文档长度
        self.N = len(self.docs)                              # 文档总数

        # 统计每个词出现在多少篇文档里（用于 IDF）
        self.df = Counter()
        for doc in self.docs:
            for word in set(doc):        # set 去重：一篇文档里算 1 次
                self.df[word] += 1

        # 预算每个词的 IDF
        self.idf = {
            word: math.log((self.N - freq + 0.5) / (freq + 0.5) + 1)
            for word, freq in self.df.items()
        }

    def score(self, query, doc_index):
        score = 0.0
        doc = self.docs[doc_index]
        tf = Counter(doc)                                  # 词在文档里的出现次数
        dl = self.doc_len[doc_index]
        for word in query.split():
            if word not in tf:
                continue
            f = tf[word]
            idf = self.idf.get(word, 0)
            # BM25 公式
            numerator = f * (self.k1 + 1)
            denominator = f + self.k1 * (1 - self.b + self.b * dl / self.avgdl)
            score += idf * numerator / denominator
        return score

    def search(self, query, top_n=3):
        # 给每篇文档打分，按分数排序
        scores = [(i, self.score(query, i)) for i in range(self.N)]
        scores.sort(key=lambda x: x[1], reverse=True)
        return scores[:top_n]


# 试一试
docs = [
    "苹果 手机 很好用",
    "我喜欢吃 苹果 香蕉",
    "苹果 公司 发布 了 新 手机",
    "今天 天气 不错",
]
bm25 = BM25(docs)
print(bm25.search("苹果 手机"))
# 输出形如：[(2, 分数), (0, 分数), (1, 分数), ...]
```

运行后你会看到：第 3 篇（"苹果 公司 发布 了 新 手机"）和第 1 篇（"苹果 手机 很好用"）分数最高，因为它们同时包含"苹果"和"手机"两个词。

## 六、有什么用？（应用场景）

| 场景 | 怎么用 BM25 |
|------|-------------|
| **网页搜索**（Google、百度早期） | 给候选网页打分排序 |
| **企业内搜 / 知识库检索** | 从文档库里找最相关的几篇 |
| **Elasticsearch / Lucene / Solr** | 这些主流搜索引擎的**默认打分算法**就是 BM25 |
| **电商搜索**（淘宝、京东） | 给商品标题/描述和搜索词打分 |
| **RAG（检索增强生成）** | 给大模型（如 ChatGPT）找参考资料时，常用 BM25 + 向量检索混合 |
| **法律 / 医学文献检索** | 精确关键词匹配的专业领域 |

> 一句话总结：**只要有"输入一段文字 → 找最相关的文档"的需求，BM25 几乎都能用。**

## 七、BM25 的优缺点

### 优点
- ✅ **简单高效**：纯数学公式，计算快，可离线预算 IDF。
- ✅ **无需训练**：不像机器学习模型需要喂数据训练，开箱即用。
- ✅ **可解释**：每个文档为什么排前面，公式拆开一看就懂。
- ✅ **关键词匹配强**：对专有名词、ID、代码等精确查询非常准。

### 缺点
- ❌ **不懂"语义"**：搜"汽车"，它找不到只写"轿车"的文档（词汇不匹配就 0 分）。
- ❌ **不懂上下文**：无法理解同义词、比喻、隐含含义。
- ❌ **中文要分词**：英文用空格切词很自然，中文得先做分词（如"苹果手机"切成"苹果"+"手机"）。

### 现代趋势：BM25 + 向量检索（混合检索）

因为 BM25 不懂语义，现在流行**两者结合**：
- **BM25** 负责精确关键词匹配（"必须出现这个词"）
- **向量检索**（如 embeddings）负责语义匹配（"意思相近也算"）

把两者的分数加权融合，效果比单独用任何一个都好。这也是当前 RAG 系统的主流做法。

---

## 八、一句话记住 BM25

> **BM25 = "词频饱和 + 词稀有度 IDF + 文档长度归一化" 三合一的文档相关性打分公式。**

三个零件各管一件事：

1. **TF 饱和**：词出现次数越多越好，但越加越少。
2. **IDF**：越稀有的词越值钱。
3. **长度归一化**：长文档要打折，长短公平。

记住这三点，你就理解了 BM25 的核心！
