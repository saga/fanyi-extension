# Bi-encoder vs Cross-encoder：搜索里的"海选"与"决赛"

> 写给零基础读者。读完你会明白：为什么搜索系统要分两步走，这两个模型各扮演什么角色。
> 本文和你根目录里那份《BM25 详解》是配套的，建议先读 BM25 那篇再看本文。

---

## 一、先讲个真实场景：HR 怎么招人？

假设你是公司 HR，收到 **10000 份简历**，要为"招一个会前端的工程师"这个岗位挑人。你会怎么做？

**你大概率会分两步：**

### 第 1 步：海选（粗筛）
拿到岗位需求后，你不会把 10000 份简历一份份仔细读。你会：
- 给每份简历**提前打好标签**：会什么技术、几年经验、学历……
- 岗位需求也提炼成几个**关键词**：前端、Vue、3 年经验……
- 用标签**快速匹配**，筛出 **50 份** 还不错的。

> 这一步追求**快**——10000 份要在几秒内过完。哪怕误判一些、漏一些也能接受。

### 第 2 步：决赛（精排）
对这 50 份，你会**仔细通读**每份简历的细节，**对照岗位需求**逐条对比："这个人虽然写了 Vue，但项目经验偏 C 端，跟我们 B 端业务契合度一般……"

> 这一步追求**准**——50 份逐一深挖，找出真正最合适的 5 个面试。

**搜索系统的逻辑和 HR 招人一模一样！** 这就是所谓的**两阶段检索（two-stage retrieval）**：

```
                第 1 阶段：召回（Recall）          第 2 阶段：重排（Rerank）
                ────────────────────────          ──────────────────────────
   1 亿篇文档  ──→  粗筛出 1000 篇  ──→  精排出 Top 10
                       (要快)                  (要准)
                       ↑                        ↑
                   Bi-encoder               Cross-encoder
                    （或 BM25）              （或 BM25 进阶版）
```

- **Bi-encoder** 干的是"海选"——又快又能处理海量文档。
- **Cross-encoder** 干的是"决赛"——又准又能捕捉细节。

接下来分别拆开讲。

---

## 二、Bi-encoder（双编码器）—— 海选选手

### 2.1 它怎么工作？

**核心思想：把查询和文档"分开处理"，各自变成一个向量，再比一比像不像。**

```
        ┌─────────────┐
查询 ──→│  编码器（Q） │──→ 向量 Q（如 [0.2, -0.5, 0.8, ...]）
        └─────────────┘                              ╲
                                                         ╲  计算相似度
                                                          ╲ （点积 / 余弦）
        ┌─────────────┐                                  ╲
文档 ──→│  编码器（D） │──→ 向量 D（如 [0.3, -0.4, 0.7, ...]）
        └─────────────┘
```

注意几个关键点：

1. **查询和文档各走各的"通道"**（所以叫 "bi"，双）。
2. 编码器把它们各自**压缩成一个固定长度的数字列表（向量）**。可以理解成"给这段文字生成一张指纹"。
3. 最后**用数学方法比两张指纹有多像**（常用点积、余弦相似度）。

### 2.2 为什么它快？（关键优势）

**因为文档的指纹可以"提前算好"！**

想想看：
- 文档库里的 1 亿篇文档，可以**离线**就把它们的向量算出来，存进数据库。
- 用户搜索时，**只需要把查询算一次向量**，然后拿这个向量和数据库里所有文档向量比对。

```
建库时（一次性，慢点没关系）：
  文档1 → 向量1 → 存入索引
  文档2 → 向量2 → 存入索引
  ...（1 亿篇慢慢算）

搜索时（要快）：
  查询 → 向量 Q → 和向量1、向量2、... 比对 → 排序返回
```

而且这种"和所有文档向量比对"的操作，有专门的**近似最近邻（ANN）算法**可以做到亚秒级搜完 1 亿篇。所以 Bi-encoder 非常适合海量初筛。

> 💡 **和 BM25 的关系**：BM25 其实也是一种"双路"思路——查询和文档各自独立处理（查询只是分词，文档也只是分词+统计），最后算个分。可以说 BM25 是"最原始版本的 bi-encoder"，只不过它用的是词频统计，而神经 bi-encoder 用的是深度学习。OpenSearch 的 neural-sparse search（也就是 bi-encoder mode）其实就是用神经网络学了一个"更聪明的 BM25"。

### 2.3 它的弱点

**查询和文档"没见面"就各自被压成了向量。**

打比方：你只看了候选人的标签卡（"会 Vue"、"5 年经验"），没让他和岗位真正"对线"。所以：
- 标签相近但实际不匹配的，可能误判为高分。
- 微妙的语义关联（比如"我做过电商后台"和"你们要 B 端经验"）可能抓不到。

所以 Bi-encoder **召回率高（能尽量捞到相关文档）但精度一般（混进来一些不太相关的）**。这正是"海选"该有的样子——宁可错杀一千，不可漏掉一个，但顺序可能不完美。

## 三、Cross-encoder（交叉编码器）—— 决赛选手

### 3.1 它怎么工作？

**核心思想：把查询和文档"拼在一起"，作为一个整体送进模型，让它们在模型内部"对话"。**

```
                ┌─────────────────────────────┐
查询 + 文档 ──→ │  编码器（同时读两段）         │──→ 一个分数（如 8.5）
                │  让查询词和文档词互相"对视"   │
                └─────────────────────────────┘
```

具体做法（以 BERT 类模型为例）：
- 输入长这样：`[CLS] 查询词1 查询词2 ... [SEP] 文档词1 文档词2 ... [SEP]`
- 模型内部的 **self-attention（自注意力）机制**让每个查询词都能"看到"每个文档词，反之亦然。
- 最后从 `[CLS]` 位置输出一个数字，就是相关性分数。

### 3.2 为什么它准？（关键优势）

**因为查询和文档在模型内部充分"交流"了。**

打个比方，Bi-encoder 像是：
- 你看了简历标签卡，也看了岗位标签卡，**各自打分**后比对。
- 缺点：你看不到"他做的电商项目"和"我们要的 B 端经验"之间那种**细致的语义关联**。

Cross-encoder 像是：
- 你**把简历和岗位需求并排放在一起**，逐条对照："这条经历→对应岗位第 3 条要求，匹配度高；这条→不太相关……"
- 优点：能捕捉到**任何细微的匹配关系**。

所以 Cross-encoder 的**精度非常高**，是当前排序质量的"金标准"。

### 3.3 它的致命弱点：慢！

**因为文档没法提前算向量！**

每来一个查询，你都得：
- 把查询和**每一个候选文档**拼起来
- 跑一遍模型

```
搜索时：
  查询 + 文档1 → 模型 → 分数1   （耗时 50ms）
  查询 + 文档2 → 模型 → 分数2   （耗时 50ms）
  查询 + 文档3 → 模型 → 分数3   （耗时 50ms）
  ...
```

如果你有 1 亿篇文档，意味着要跑 1 亿次模型——**完全不可行**（按 50ms/次算，要 60 多天！）。

> 💡 这也是为什么 OpenSearch 文档里写：cross-encoder 的用途是 **"query reranking"（查询重排）**——它不是用来做初筛的，而是用来给初筛结果"二次排序"的。

## 四、为什么用在不同的地方？（核心问题）

现在答案应该很清楚了：

| | Bi-encoder | Cross-encoder |
|---|---|---|
| **角色** | 海选 / 召回 | 决赛 / 重排 |
| **速度** | 极快（查询只算 1 次向量） | 很慢（每个文档都要算） |
| **精度** | 一般 | 极高 |
| **能否预计算文档向量** | ✅ 能（这是快的根本原因） | ❌ 不能（必须查询+文档一起算） |
| **处理文档量** | 1 亿级 | 通常 100~1000 篇 |
| **OpenSearch 用途** | 神经搜索、向量检索、neural-sparse | Reranking API（搜索结果重排） |

### 典型流水线

```
用户查询
   │
   ▼
┌────────────────────────────────────┐
│ 第 1 阶段：Bi-encoder（或 BM25）   │  ← 快速从 1 亿篇捞 1000 篇
│ - 查询 → 向量                       │
│ - 和预存的文档向量比对              │
└────────────────────────────────────┘
   │ 1000 篇候选
   ▼
┌────────────────────────────────────┐
│ 第 2 阶段：Cross-encoder 重排      │  ← 精排出 Top 10
│ - 查询+文档 → 模型 → 分数           │
│ - 对 1000 篇逐一精算                │
└────────────────────────────────────┘
   │ 最终 Top 10
   ▼
展示给用户
```

**为什么必须这么分工？**

- 如果**只用 Cross-encoder**：太慢，根本搜不动 1 亿文档。
- 如果**只用 Bi-encoder**：速度够快，但排序精度差，前几名可能不太相关。
- **两者结合**：Bi-encoder 保证"快速 + 召回全"，Cross-encoder 保证"前几名精准"。各取所长。

> 这就像 HR 招人：先用标签卡海选 50 人（不然你看不完 10000 份简历），再当面深聊挑 5 个（不然容易看走眼）。

## 五、回到 OpenSearch 的具体语境

你提到的两篇文档，正好对应这两个角色：

### Bi-encoder 出现在 neural-sparse 文档里

OpenSearch 的 **neural sparse search**（神经稀疏搜索）是一种**改进版的 BM25**。它有几种模式：

- **Doc-only 模式（默认）**：文档用神经网络编码成稀疏向量（一堆 `<词: 权重>`），查询只用一个轻量的分词器处理。**速度最快**。
- **Bi-encoder 模式**：文档和查询**都用神经网络编码**。**更准但更慢**（查询也要跑模型）。

> ⚠️ 注意：OpenSearch 文档里说的 "bi-encoder" 是个**更具体的术语**——它指"文档和查询都用同一个神经网络编码器处理"，是和 doc-only 模式（查询侧只用轻量分词器）相对的概念。
>
> 但无论哪种模式，它们都属于**广义的 bi-encoder 范式**：查询和文档**分开编码**，然后比对。这点和本文讲的大原则一致。

所以 bi-encoder mode 更准的原因很直观：查询侧也用了"聪明的神经网络"而不是"傻瓜分词器"，捕捉到的语义更丰富。

### Cross-encoder 出现在 pretrained-models 文档里

OpenSearch 文档原话：

> "Cross-encoder models support **query reranking**."

意思就是：cross-encoder 模型**只用来做重排**。流程是：
1. 你先用任意方法（BM25 / neural search / k-NN）搜出 Top N（比如 100 篇）。
2. 再调 cross-encoder 对这 100 篇重新打分排序。

OpenSearch 给的 cross-encoder 调用示例长这样（你给一个查询 + 几个文档，它返回每个文档的相关性分数）：

```
查询：   "today is sunny"
文档：   ["how are you", "today is sunny", "today is july fifth", "it is winter"]

返回分数：[-6.08, 10.22, -1.40, -4.59]   ← 完全匹配的那个分数最高
```

看，它**一次性吃查询 + 多个文档**，输出每个的相关性分数。这就是典型的 cross-encoder 用法——给候选列表"精排"。

## 六、一张图总结区别

```
┌─────────────────────── Bi-encoder ───────────────────────┐
│                                                           │
│   查询 ──→ [编码器] ──→ 向量 Q ──┐                        │
│                                   ├─→ 相似度 ──→ 分数     │
│   文档 ──→ [编码器] ──→ 向量 D ──┘                        │
│                                                           │
│   ✅ 文档向量可预存 → 极快 → 适合海量初筛                 │
│   ❌ 查询/文档没"见面" → 精度有限                         │
│                                                           │
│   类比：看标签卡快速筛选简历                              │
└───────────────────────────────────────────────────────────┘

┌─────────────────────── Cross-encoder ────────────────────┐
│                                                           │
│   [查询 + 文档] ──→ [编码器（一起读）] ──→ 分数           │
│                                                           │
│   ✅ 查询/文档充分交互 → 极准                             │
│   ❌ 每对都要算 → 极慢 → 只能用于 Top-K 重排              │
│                                                           │
│   类比：把简历和岗位并排细读                              │
└───────────────────────────────────────────────────────────┘
```

## 七、一个生活化的对比例子

假设你在相亲平台找对象，平台有 100 万用户。

| | Bi-encoder 思路 | Cross-encoder 思路 |
|---|---|---|
| 做法 | 给每个人打"标签卡"（年龄、爱好、职业），你的要求也做成标签卡，快速匹配 | 把你和每个候选人的资料**摊在一起仔细读**，逐条对照 |
| 速度 | ⚡ 1 秒筛 100 万人 | 🐢 1 个人要聊半小时 |
| 准确度 | 中等，可能错过微妙匹配 | 极高，能发现深层契合 |
| 用在哪 | 第一轮筛选（挑出 100 个） | 最终决策（从 100 个里选 3 个见面） |

真实平台就是这么干的：先用算法（bi-encoder 类）推一波，再让你详细看资料（cross-encoder 类）做最后选择。

## 八、和 BM25 的关系（连接上一篇文档）

| 特性 | BM25 | Neural Bi-encoder (sparse/dense) | Cross-encoder |
|------|------|----------------------------------|---------------|
| 查询/文档处理方式 | **独立**（查询分词、文档分词） | **独立**（各编码成向量） | **联合**（一起送进模型） |
| 能否预计算文档表示 | ✅ 能（倒排索引） | ✅ 能（预存向量） | ❌ 不能 |
| 速度 | ⚡⚡⚡ 极快 | ⚡⚡ 快 | 🐢 慢 |
| 精度 | 中 | 中高 | 高 |
| 是否懂语义 | ❌（纯关键词） | ✅（神经网络） | ✅✅（深度交互） |
| 典型用途 | 初筛 | 初筛 | 重排 |

可以看到：
- **BM25 和 bi-encoder 同属"独立处理"阵营**，都适合初筛，只是 BM25 是关键词层面、bi-encoder 是语义层面。
- **Cross-encoder 是另一阵营**——"联合处理"，精度碾压但速度拉胯，只能做重排。

OpenSearch 的 **neural sparse search** 本质是"**用神经网络重做的 BM25**"——既保留了 BM25 那种独立编码、可倒排索引的速度优势，又加上了神经网络的语义理解能力。所以它和 BM25 是同位竞争关系（都做初筛），而 cross-encoder 是它们的"下游精排工"。

## 九、实战中的常见组合

现代搜索/RAG 系统的"标准配方"：

### 配方 1：经典两阶段
```
BM25 / Neural Bi-encoder  ──→  Cross-encoder 重排  ──→  最终结果
   (Top 1000)                  (Top 10)
```

### 配方 2：混合召回（Hybrid Retrieval）+ 重排
```
        ┌─→ BM25 召回 ─────────┐
查询 ───┤                       ├─→ 合并去重 ──→ Cross-encoder 重排 ──→ 结果
        └─→ 向量 Bi-encoder ───┘
        （关键词召回）           （语义召回）
```

> 这也是当前 **RAG（检索增强生成，比如给 ChatGPT 喂资料）** 的主流做法：用多种召回方式捞全，再用 cross-encoder 精排，最后把最相关的喂给大模型。

### 配方 3：极简（只用一阶段）
- 小数据量 / 对延迟极敏感 → 只用 BM25 或只用 Bi-encoder。
- 精度要求极高 / 数据量小 → 只用 Cross-encoder。

---

## 十、一句话记住它们

> **Bi-encoder 是"相亲平台的标签匹配"——快但粗，用来海选；**
> **Cross-encoder 是"两人面对面深聊"——慢但准，用来决赛。**
>
> **它们不是二选一，而是配合使用：海选靠 Bi-encoder（或 BM25），决赛靠 Cross-encoder。**

## 附：术语速查表

| 术语 | 含义 |
|------|------|
| **Encoder（编码器）** | 把文字转成向量（一串数字）的模型 |
| **Embedding（嵌入/向量）** | 文字被编码后的数字表示，是它的"语义指纹" |
| **Bi-encoder（双编码器）** | 查询和文档各自编码，再比对的架构 |
| **Cross-encoder（交叉编码器）** | 查询+文档一起送进模型，直接出分数的架构 |
| **召回（Recall）** | 第一阶段，从海量文档里粗筛 |
| **重排（Rerank）** | 第二阶段，对召回结果精排 |
| **Self-attention** | Transformer 内部机制，让序列中每个词都能"看到"其他词 |
| **Sparse vector（稀疏向量）** | 大部分维度是 0 的向量（像 BM25 的词项权重） |
| **Dense vector（稠密向量）** | 每个维度都有值的向量（如 768 维浮点数） |
